## 1. 产品定位 vs 开发计划：匹配度分析

### 1.1 逐条对照

| 产品定位要素 | 开发计划是否覆盖 | 状态 | 备注 |

| 每天从 RSS 获取 | ✅ | ✅ | Phase 2 包含 RSS 采集器 |
| 从 arXiv 获取 | ✅ | ✅ | Phase 2 包含 arXiv 采集器 |
| 从 GitHub 获取 | ✅ | ✅ | Phase 2 包含 GitHub Search 采集器 |

### 1.2 关键偏差：缺少"学习"属性

你的产品定位强调的是 **"中文学习卡片"**，但开发计划中用的是 **"情报卡片"**。

```
产品定位：学习卡片 → 面向"学习者"，强调"理解"
开发计划：情报卡片 → 面向"信息获取者"，强调"浏览"
```

**差异体现：**

| 学习卡片（定位） | 情报卡片（计划） |
|----------------|----------------|
| 核心要点 | ✅ 有 |
| 适合谁关注 | ❌ 缺失 |
| 相关术语解释 | ❌ 缺失 |
| 可能影响 | ❌ 缺失 |
| 推荐理由 | ✅ 有（但放在 Phase 3，不是核心） |

**建议**：在 Phase 1 的数据模型中就加入 `audience`、`glossary`、`impact` 字段，确保每条内容都是一张"可学习的信息卡"，而不只是"带摘要的新闻"。

## 2. 阶段划分：逻辑分析

### 2.1 当前顺序

```
Phase 0: 项目初始化
Phase 1: 数据库 + 基础页面（mock 数据）
Phase 2: 采集器
Phase 3: AI 处理管线
Phase 4: AI 助手
Phase 5: 部署
Phase 6: 后续增强
```

### 2.2 问题分析

| 问题 | 说明 | 影响 |
|------|------|------|
| **Phase 1 用 mock 数据太多** | 先做 mock 页面再接数据库，可能浪费时间 | 中 |
| **Phase 0 和 Phase 1 可合并** | 初始化 + 基础页面可以一起做 | 低 |
| **Phase 2 和 Phase 3 可部分并行** | 采集器写完一个就可以开始接 AI 处理 | 中 |
| **Phase 4 (AI 助手) 被放到最后** | 这是产品的核心差异化功能，应该更早验证 | 高 |
| **缺少"端到端验证"阶段** | 应该在 Phase 2-3 之间有一个"跑通全流程"的节点 | 高 |

### 2.3 建议优化顺序

```
Phase 0: 项目初始化 + 基础页面（合并原 Phase 0 和 1）
         → 有 mock 数据的可交互页面

Phase 1: 数据库 + 数据模型（包含"学习"字段）
         → 页面对接真实数据库

Phase 2: 核心采集器（先只做 RSS，一个源即可）
         → 端到端验证：抓取 → 存储 → 展示

Phase 3: AI 处理管线
         → 端到端验证：抓取 → LLM 处理 → 中文卡片展示

Phase 4: 补全采集器（arXiv + GitHub）
         → 扩大数据源覆盖

Phase 5: AI 助手（核心差异化）
         → 用户可以追问

Phase 6: 部署 + 打磨
         → 上线
```

**关键变化：**
1. 合并 Phase 0 和 1，减少空转
2. 先用一个 RSS 源跑通全流程，再扩展
3. AI 助手提前到部署前，确保核心功能先验证
4. 每个 Phase 结束都有一个"可演示的端到端流程"

## 3. 技术选型：风险评估

### 3.1 正确的选择 ✅

| 决策 | 评价 |
|------|------|
| Next.js + React | ✅ 前后端一体，适合 MVP |
| Tailwind + shadcn/ui | ✅ 快速搭建工具型界面 |
| Supabase PostgreSQL | ✅ 免费额度够用，后续可扩展 |
| PostgreSQL 全文搜索 | ✅ MVP 不需要向量库 |
| MCP 后置 | ✅ 先跑通再标准化 |

### 3.2 需要注意的风险 ⚠️

| 决策 | 风险 | 建议 |
|------|------|------|
| **Prisma 或 Drizzle 二选一** | ORM 选择不确定可能导致返工 | **建议直接选 Drizzle**：更轻量、TypeScript 原生、与 Supabase 配合更好 |
| **Vercel Cron 每天 1 次** | 免费版限制：1 个 cron job，每天 1 次，60 秒内完成 | 如果采集+LLM 处理超过 60 秒，需要用 **GitHub Actions** 补充 |
| **AI 调用"OpenAI/Claude/兼容 API"** | 太模糊，实际开发时需要明确 | 建议：**先用 OpenAI API**（生态最成熟），预留切换接口 |
| **Supabase 免费额度** | 500MB 数据库 + 500MB 文件 + 50k MAU | 新闻数据增长快，注意监控用量 |

### 3.3 建议补充的技术决策

| 缺失项 | 建议 |
|--------|------|
| **RSS 解析库** | 推荐 `rss-parser`（Node.js）或 `feed`（更轻量） |
| **arXiv API** | 使用 `arxiv` npm 包或直接调用 `export.arxiv.org/api/query` |
| **GitHub API** | 使用 `octokit` 或直接调用 REST API |
| **LLM 输出解析** | 推荐用 `zod` 定义 schema，配合 `openai` SDK 的 `response_format` |
| **错误处理** | 每个采集器需要独立的 try-catch，失败不影响其他源 |

### 4.2 缺少 Prompt 设计

Phase 3 提到"设计摘要 prompt"，但没有给出具体设计。建议在开发前就写好：

```markdown
## Prompt 设计模板

### 摘要生成 Prompt

你是一个 AI 领域资讯分析师。请分析以下内容，输出 JSON 格式：

输入信息：
- 标题：{title}
- 来源：{source}
- 摘要/正文：{content}

请输出：
{
  "title_cn": "中文标题（不超过30字）",
  "summary_cn": "80-120字中文摘要",
  "key_points": ["要点1", "要点2", "要点3"],
  "tags": ["标签1", "标签2", "标签3"],
  "category": "news|paper|tool|project|opinion",
  "importance": 1-5,
  "reason": "为什么值得看（30字以内）",
  "audience": ["适合谁看"],
  "glossary": [{"term": "术语", "explain": "解释"}]
}

规则：
1. 如果内容与 AI 无关，importance 设为 1
2. 重要性评分标准：5=重大突破，4=重要进展，3=值得关注，2=一般动态，1=不太相关
3. 摘要要客观，不夸大
```

### 4.4 缺少错误处理策略

```markdown
## 错误处理策略

| 场景 | 处理方式 |
|------|---------|
| 单个 RSS 源失败 | 跳过该源，继续其他源，记录日志 |
| LLM API 调用失败 | 重试 2 次，间隔 5 秒；仍失败则标记为 pending |
| LLM 输出格式错误 | 尝试重新生成 1 次；仍失败则存储原始内容 |
| 数据库写入失败 | 重试 1 次；仍失败则记录到错误队列 |
| 采集任务超时 | 中断当前任务，保留已处理的数据 |
```

```markdown
## 关键验收测试

### 采集流程

- [ ] 再次触发采集 → 不会重复入库
- [ ] 模拟 RSS 源失败 → 其他源正常工作

### AI 处理
- [ ] 英文内容 → 生成中文摘要
- [ ] 中文内容 → 生成中文摘要
- [ ] 非 AI 内容 → importance=1，不展示在首页
- [ ] LLM 返回非法 JSON → 自动重试

### AI 助手
- [ ] 问"今天有什么重点" → 返回 importance≥4 的内容
- [ ] 问"某条内容讲了什么" → 返回该条的详细解读
- [ ] 问无关问题 → 礼貌拒绝，不编造内容
```

## 7. 结论

你的开发计划**框架正确、范围合理**，但在两个方面需要调整：

1. **产品定位的"学习"属性**没有充分体现——每条内容应该是一张"帮助你理解的学习卡片"，而不只是"带摘要的新闻卡"
2. **开发顺序**可以优化——先跑通一个源的全流程，再扩展；AI 助手作为核心差异化功能应该提前验证

按建议调整后，这份计划可以很好地支撑你的产品定位。
